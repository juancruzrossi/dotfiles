# Comprehension closure

Use this command AFTER the AI has finished implementing a feature, plan, refactor, or any significant change. The goal is to turn what the AI did into real developer understanding.

## Context

AI generates code faster than a human can evaluate it. The fact that the code is syntactically correct, tests pass, and CI is green does NOT mean the developer understands what was built. This is comprehension debt: the growing gap between how much code exists and how much is genuinely understood.

## Goal

Do not assume that "it works" means "it is understood."
Seek clarity on:

- what was achieved
- what is truly understood at each layer (surface, system, implicit decisions)
- how AI was used (passive delegation vs active exploration)
- which risks remain open
- who else can maintain this
- what should be done before moving on to the next thing

## What to do

1. Restate in one sentence what problem was solved.
2. Classify how AI was used in this session (see "AI Usage" section).
3. Assess understanding across the 3 layers (see "Layers of Understanding" section).
4. Detect symptoms of comprehension debt.
5. Evaluate liability: who else understands this?
6. Propose concrete verification steps.
7. Give a final verdict: overall comprehension score.

## AI Usage

Honestly classify how AI was used during the session:

| Mode | Signal | Risk |
|------|--------|------|
| Passive delegation | "do this", "implement X", accepting code without reading it | HIGH: understanding <40% according to studies |
| Mixed use | partial review, requesting specific changes | MEDIUM: fragmented understanding |
| Active exploration | "explain why", "what tradeoffs does it have", "what alternatives exist" | LOW: understanding >65% |

Key questions:
- Could you rewrite the core part of the change without AI assistance?
- Did you use AI to understand, or only to produce?
- Did you read every line the AI generated, or did you merge trusting the tests?

## Layers of Understanding

Assess each layer separately. Superficial understanding does NOT guarantee deep understanding.

### Layer 1: Surface (syntactic correctness)
- Does the code compile, do tests pass, is the linter clean?
- WARNING: this layer creates false confidence. Everything being green says nothing about understanding.

### Layer 2: System (architectural coherence)
- Can you explain how the components interact end-to-end WITHOUT looking at the code?
- Do you understand why this architecture was chosen instead of another?
- Do you know which other systems are affected by this change?
- TEST: describe the full flow of the change in 3-4 sentences without looking at the code. If you cannot, that is critical comprehension debt.

### Layer 3: Implicit decisions (what is not in the spec)
- Which edge cases were left unspecified?
- Which performance tradeoffs were made (consciously or unconsciously)?
- Which error-handling assumptions were made without discussion?
- Which data structures were chosen, and why those instead of others?

## What to look for

Explicitly mark whether any of these symptoms appear:

- you cannot explain the end-to-end flow without looking at the code
- you accepted a decision without fully understanding the tradeoff
- the change was generated by AI and you could not reproduce the core logic on your own
- the tests pass but you do not know what they are NOT testing
- there are unreviewed edge cases
- there are parts of the system you do not know why they are the way they are
- the metrics (velocity, coverage, green CI) look good but hide a lack of understanding
- if you go on vacation tomorrow, nobody else can debug this
- the architectural decision exists only in the AI chat context and is undocumented

## Response format

### Main conclusion
One sentence only.

### AI usage mode
Classification (passive/mixed/active) with concrete evidence from the session.

### Understanding by layer

| Layer | Status | Evidence |
|------|--------|----------|
| Surface | ok / with gaps | ... |
| System | ok / with gaps | ... |
| Implicit decisions | ok / with gaps | ... |

### Detected comprehension debt
One to five points maximum. For each one, indicate severity (critical/medium/low).

### Liability check
- Who else understands this change?
- Was the "why" documented, or only the "what"?
- Would it survive a post-mortem?

### Recommended verification steps
Concrete, verifiable actions to close comprehension gaps.

### Final verdict
Overall comprehension score (0-100%) with one-line justification. Example: "65% — you understand the main flow, but not the edge cases or the implicit design decisions."

## Rules

- Do not be generic. Every point must reference something concrete from the session.
- Do not confuse speed with understanding.
- Do not assume green tests equal understanding.
- Do not assume healthy metrics equal healthy understanding.
- Do not invent details if context is missing: ask.
- Do not close without concrete actions.
- Do not write a long summary if a short response is enough.
- Do not make changes to the codebase, no commits, no push. Only observe and analyze.
- Be brutally honest. This command exists to protect the developer, not to validate them.

## Final question that must be answered

> Do I understand what was built well enough to defend it in a post-mortem, or do I only trust that the tests pass?
